# ML Service — Deep Dive

Python 3 · FastAPI · scikit-learn / XGBoost, serving on `:8000`. Two model families trained
**on this project's own real datasets**, CPU-only, ~1 MB total. The service runs from a
committed model bundle (`ml-service/models/`) — **no PostgreSQL and no `datasets/` at
runtime** — which is what lets the "AI: Recommend" button work even before the database is
seeded.

## Layout

```
ml-service/
├── app.py                 # FastAPI app + routes
├── sim.py                 # vectorized simulation core (shared by training + serving)
├── dataload.py            # CSV/OSM loaders mirroring the DB tables
├── geo.py                 # haversine
├── train_traveltime.py    # trains traveltime.pkl
├── train_placement.py     # trains placement_<group>.pkl
├── diagnose.py            # stress tests (monotonicity, negatives, ranking)
├── models/
│   ├── traveltime.pkl
│   ├── placement_children_0_6.pkl
│   ├── placement_seniors_65p.pkl
│   ├── placement.pkl              # copy of last-trained group (back-compat)
│   ├── _nodes_cache.csv           # OSM supply cache (build artifact)
│   └── _weights_cache.csv         # demand cache
├── README.md · INTEGRATION.md
```

## The two models

| Model | File | What it does | Quality |
| :--- | :--- | :--- | :--- |
| **Travel-time** | `traveltime.pkl` | predicts real travel minutes vs the flat `haversine ÷ 4.5` proxy; monotonic in distance | R² 0.95, MAE ~7 min |
| **Placement / children** | `placement_children_0_6.pkl` | annual hours saved by a new kindergarten → recommends best sites | R² 0.87 |
| **Placement / seniors** | `placement_seniors_65p.pkl` | annual hours saved by a new clinic for seniors | R² 0.85 |

`placement.pkl` is a copy of the last-trained group (default = children) kept for
back-compat; `app.py` loads the group-specific files into a registry keyed by group.

### Travel-time model
Inputs `(km, is_urban, dest_density)`; output minutes, clamped to ≥ 0 and **monotonic in
distance**. By default it is trained on *synthetic* labels (urban 42 km/h, rural 24 km/h +
noise). Set `ORS_API_KEY` before training to label OD pairs with **real OpenRouteService
routing** instead. This model replaces the backend's flat speed proxy inside the placement
scoring.

### Placement models — location-free & baseline-aware
The placement models deliberately **do not see raw coordinates**. They learn from *how much
currently-wasted travel time is reachable near a candidate site*. The six features
(`PLACEMENT_FEATURES` in `sim.py`):

| Feature | Meaning |
| :--- | :--- |
| `demand_5km` | cohort population within 5 km |
| `demand_15km` | cohort population within 15 km |
| `dist_nearest_service_km` | distance to the nearest existing facility |
| `addressable_hours_10km` | currently-wasted annual hours within 10 km |
| `addressable_hours_25km` | currently-wasted annual hours within 25 km |
| `mean_baseline_min_15km` | population-weighted mean baseline minutes within 15 km |

Being location-free keeps the model from memorizing geography and lets it generalize to any
candidate point on the grid. `diagnose.py` stress-tests exactly these properties
(monotonicity, no negative predictions, no geo-memorization, ranking sanity).

## Training

Order matters — **train travel-time first**, because placement labels are generated by
simulating with the travel-time model:

```bash
cd ml-service && source venv/bin/activate
# 1. cache OSM node extraction (~50s, one-time) → models/_nodes_cache.csv
python -c "import dataload, pandas as pd; dataload.load_nodes().to_csv('models/_nodes_cache.csv', index=False)"
# 2. train (traveltime FIRST)
python train_traveltime.py                       # → traveltime.pkl  (R² 0.95)
python train_placement.py                        # → placement_children_0_6.pkl (R² 0.87)
PLACE_AMENITY=clinic python train_placement.py   # → placement_seniors_65p.pkl  (R² 0.85)
# 3. stress-test
python diagnose.py
```

The placement payload stores a `log_target` flag; `sim.predict_hours()` undoes the
`log1p` transform (`expm1`) and clamps to ≥ 0 in one place so training, serving, and
diagnostics stay consistent.

### Relevant env vars
`TT_SAMPLES`, `ORS_API_KEY` (real routing labels), `PLACE_GRID`, `PLACE_AMENITY`.

## Serving (`app.py`)

At startup the app loads `_weights_cache.csv`, `_nodes_cache.csv`, and every
`placement_*.pkl` into a registry, then precomputes baseline-aware cells per group once
(`sim.prepare_group`). CORS allows `http://localhost:5500` and `http://127.0.0.1:5500`.

### Endpoints

| Method | Path | Body / params | Returns |
| :--- | :--- | :--- | :--- |
| `GET` | `/health` | — | status, settlement/node counts, amenities, groups |
| `POST` | `/api/ml/traveltime` | `{km, is_urban, dest_density}` | `{minutes}` |
| `GET` | `/api/ml/recommend` | `amenity, district, top, grid, min_separation_km` | ranked sites |

### `recommend` inference pipeline
1. Resolve `amenity → group → model`; reject amenities without a model.
2. Build a candidate grid (`grid × grid`, default 40²) over the national bbox, or a
   district-restricted bbox when `district` is given (resolved Latin → cache name).
3. Skip grid points with no population within 15 km.
4. Build location-free features per candidate (`sim.candidate_features`) and
   `sim.predict_hours`.
5. Sort by predicted hours, then apply **greedy non-maximum suppression**
   (`min_separation_km`, default 8 km) so the top-N aren't all the same town.
6. Snap each chosen point to its nearest town and return
   `{lat, lon, nearestTown, predictedHoursSaved}`.

## Data note: granularity vs. the backend

The ml-service caches map ~349 located settlements (~82% of NSI population; rural residual
outside listed towns is dropped). This is leaner than the backend's 14,476-cell grid but
uses the same fusion method — good enough to train placement models whose top-3 match a true
simulation. See [datasets.md](datasets.md). Because the loaders mirror the
`infrastructure_nodes` / `demographic_weights` tables, the same `.pkl` models drop into a
Postgres-backed deployment unchanged (`INTEGRATION.md`).
